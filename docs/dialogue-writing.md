# 对白写作与去重

这份文档说明游戏对白的重复控制、角色口吻和中文表达要求。新增剧情、条件分支或生成式文案时，应先判断台词在当前周目中的作用，再决定能否重复。

## 基本目标

对白需要同时满足三个目标：

1. 玩家能听出说话人是谁。
2. 每句话都和当前场景、关系状态或研究冲突有关。
3. 同一周目内不会无意重复完整台词，也不会连续表达同一个意思。

句子数量越多，体验未必越好。真正需要增加的是场景差异、信息差异和人物反应差异。只替换几个同义词，玩家仍会察觉内容在循环。

## 重复内容的处理规则

### 长对白

完整的剧情对白默认在一条游玩线路中只出现一次。条件持续满足时，路线方案可以继续保留，负责引出路线的对白应停止重复。

项目中的条件分支按以下方式处理：

- 整个分支只需触发一次时，设置 `once: true`。
- 路线方案需要长期保留时，可以持续命中分支，并通过 `contribute.setFlags` 记录引导对白已经出现。
- 当 `contribute.setFlags` 中的值已经写入状态，运行时会隐藏该分支的旧对白，同时保留研究方案和提示覆盖。

这种处理可以避免玩家每个月重新听一遍同样的关心、警告或邀请，也不会让已经解锁的路线突然消失。

### 回调台词

有意回收前文时，可以重复关键词、动作或意象。回调需要带来新的关系信息或情绪变化，不能整句照搬。

例如，早餐可以在不同阶段承担不同作用：

- 初期表示她记得你的习惯。
- 中期提醒你遵守休息承诺。
- 后期用早餐是否出现表现信任变化。

物件可以重复，句子和情绪结果需要变化。

### 关系升温节点

关系值达到门槛只说明角色愿意进一步靠近，不能自动生成一段通用夸奖。关系节点需要对应具体事件，例如兑现承诺、共同修正错误、展示失败记录、公开支持或交出一项重要判断。

2025 年使用三位角色各自的行为分支，不再额外插入通用亲密台词。2023、2024 的补充关系节点固定在特定月份，每个节点只出现一次，并使用不同的动作、信息和关系结果。

删除角色姓名后，如果一句关系台词可以原样交给其他角色使用，通常说明内容还不够具体。没有新事实、新权限或新风险时，可以不插入关系对白。

### 角色口头习惯

角色可以保留少量稳定表达，例如林若宁会要求补证据链，陈星禾会追问归因，周明昭会先画风险边界。

口头习惯应保留思维方式，避免固定成完整台词。连续几话使用同一开头、同一比喻和同一结论，会让角色像在调用模板。

### 系统提示

短功能提示可以重复，例如 `点击继续。`、`上一句` 和 `档案与研究室`。这类文字负责操作确认，稳定比变化更重要。

剧情提示、角色反馈和结算评语需要更高的多样性。它们承担叙事功能，重复会直接暴露内容模板。

## 两类重复都要检查

### 完全重复

完全相同的长句最容易发现，也最适合自动检查。检查前可以统一空格、标点和换行，再比较文本。

需要重点检查：

- 场景节点的 `text`
- 研究方案的 `description`
- 月度结算的角色评语
- 条件分支追加的对白
- 同一年份十二话之间的开场句和任务说明

### 语义重复

换了词仍在表达同一个意思，也会造成疲劳。例如连续几个月都用早餐、脸色、熬夜和长跑提醒玩家休息，即使句子不同，玩家仍会觉得角色只会说这一件事。

人工复核时应同时记录每句话的功能：

- 提供新事实
- 追问证据
- 表达关心
- 改变关系
- 提醒风险
- 回收承诺
- 推进选择

同一角色连续两次承担相同功能时，应改变场景动作、信息来源或情绪结果。

## 角色之间要有表达差异

### 林若宁

她从生活细节和研究过程切入。关心通常通过便签、早餐、文件修改和陪伴复盘体现。

她可以直接提醒身体状态，金融比喻只需点到为止。关心场景里堆叠模型、回撤、框架和长跑，会削弱真实感。

参考表达：

> 林若宁把早餐推到你手边。
>
> 这周你已经第三次半夜发研报了。我担心你先把身体熬垮。先吃点东西。
>
> 她翻到你新改的目录。
>
> 这版框架确实比上个月利落。

### 陈星禾

她从数据异常和验证过程切入。语速可以快，术语必须对应当前图表或模型问题。

她的情绪变化可以通过关掉屏幕、删掉曲线、重新跑样本和主动暴露失败记录表现。不要每次都靠 `Alpha`、拥挤度和信噪比制造角色感。

### 周明昭

她先看长期代价，再看短期收益。语言较短，停顿较多，提醒通常带有清晰边界。

参考表达：

> 周明昭看了你两秒，把电脑屏幕按灭。
>
> 你脸色很差。研究要做很多年，今晚先停。
>
> 她把电源线从你手边收走。
>
> 再熬一夜也不会让错误的判断变对。

### 赵承宇

他从同级协作和可复算的数据切入。语气可以更口语化，分歧之后仍要留下继续合作的空间。

他不承担导师式总结。需要讲知识时，让他指出具体样本、口径或回测问题，再由玩家完成判断。

## 中文表达要求

### 先写场景，再写观点

角色开口前应有一个能被看见的动作，例如合上电脑、圈出数据、推来早餐或擦掉白板。动作需要服务情绪，避免每句话都先盯着你看几秒。

观点尽量落在当前事件上。抽象总结适合放在结算或章节收束，日常对白更适合具体问题。

### 动作与对白分段

动作描写和角色说出口的话分别成段。界面通过空行识别段落，内容使用 `dialogueText(...)` 组合。动作不放在括号里，对白不额外添加引号。角色姓名已经由界面显示，无需再用符号重复标记。

一段场景可以按动作、对白、动作、对白的顺序推进：

> 林若宁把你的结论圈出来。
>
> 方向可能是对的，可中间这几步还是空的。
>
> 她看着你，没有退开。
>
> 我相信你。证据缺口还在，这几步必须由你补出来。

每个动作都应改变视线、物件、距离或下一句话的含义。单独拆段却没有节奏作用，只会让对白框变得更长。

### 多人观点分别呈现

同一节点出现多位角色时，每位角色需要独立的展示单元。观点交锋应分别显示姓名、方法标签和观点正文，避免生成类似以下结构的长句：

> 林若宁的基本面说：……，陈星禾的量价说：……，周明昭的风控说：……。

这类拼接会让说话人、停顿和视线变化混在一起，也容易产生连续句号。年度剧情使用三张观点卡，档案抽屉也按角色保留三条观点。新增多人场景时，应先确认界面是否支持逐人呈现，再决定内容结构。

### 控制术语密度

一段对白通常保留一个核心术语。必须同时出现多个术语时，应让角色逐步解释它们之间的关系。

常用缩写可以保留，例如 `AI`、`SaaS`、`ARR`、`Alpha` 和 `Barra`。中文句子中的英文与中文之间保留空格。能用自然中文说明时，优先写中文，例如把 `miss` 改为业绩不及预期。

### 减少翻译腔

优先使用短句和常见动词。删去不承担信息的铺垫、口号和抽象名词。

需要重点检查的表达包括：

- 为了强调而先否定一次，再重复真正结论
- 连续使用这意味着、这代表着、从某种意义上说
- 把人物对白写成课程定义
- 每个人都用同一种完整对偶句
- 为了显得专业而加入无关英文

### 使用中文标点

中文说明和对白使用中文逗号、句号、冒号、问号和括号。代码、路径、参数和字段名保留行内代码格式。

省略语气优先通过停顿和短句完成。连续省略号、感叹号和括号补充会让台词显得像翻译后的轻小说脚本，应谨慎使用。

内容字段与界面收束语拼接前，需要先统一句末标点。完整句子由内容字段自行保留一个句号，界面层不要再追加句号。列表标签、人物姓名和方法名称不加句号。自动检查应覆盖重复句号、英文逗号、英文括号和多人台词拼接模板。

## 生成式文案的复核流程

生成对白后按以下顺序检查：

1. 按角色汇总本周目可能出现的全部 `text`。
2. 查找完全相同的长句和高度相似的句子。
3. 标记每句话承担的叙事功能。
4. 检查同一角色是否连续使用相同动作、比喻和句式。
5. 删除没有推动事实、关系或选择的信息。
6. 检查术语是否能被场景中的人物自然说出口。
7. 检查动作和对白是否分段，正文是否滥用括号或引号。
8. 检查中英文空格、中文括号和标点。
9. 检查多人场景是否按说话人分开，句末是否出现重复标点。
10. 实际连续游玩十二话，观察回调是否自然。

自动检查可以发现完全重复和固定句式。语义重复、角色失真和情绪节奏仍需要人工通读。生成模型很擅长把同一意思重新包装，人类也很擅长在第三次包装时失去耐心。

仓库级文本检查可以单独运行：

```bash
uv run python scripts/validate_text_quality.py
```

该检查覆盖说明文档，以及 `src/game/` 下的版本化剧情 JSON 和 TypeScript 字符串。它会识别重复句末标点、中文语境中的英文标点与括号，以及用逗号动态拼接多位说话人的代码。TSX 界面文案仍需要在组件测试和人工通读中确认。

## 提交前检查

修改对白或分支后，至少确认以下事项：

- 长剧情对白在同一线路中没有无意重复。
- 持续生效的路线只重复方案，不重复引导对白。
- 有意回调包含新的信息或情绪结果。
- 关系升温来自具体事件，不能只靠关系值生成通用夸奖。
- 动作和对白分别成段，不用括号和引号重复标记。
- 多人观点按角色分别展示，不把姓名、方法和台词拼成一个长句。
- 中文句末只有一个标点，界面层不会重复补句号。
- 三位导师对同一事件使用各自的方法和语言。
- 英文术语确有必要，并符合中文排版。
- 台词读起来像具体场景中的交流。
- `npm run test:run` 和 `npm run check` 通过。
